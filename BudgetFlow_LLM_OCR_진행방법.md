# BudgetFlow LLM/OCR 진행방법

## 역할 요약

사용자가 보낸 텍스트·이미지를 **정형화된 지출 데이터**로 변환하는 분석 엔진을 담당한다.
백엔드 Step Functions에서 Lambda로 호출되며, 분석 결과와 신뢰도를 반환한다.

---

## 기술 스택

| 영역          | 기술                           | 용도                                      |
| ------------- | ------------------------------ | ----------------------------------------- |
| LLM           | Amazon Bedrock (Claude)        | 텍스트 파싱, 카테고리 분류, OCR 결과 정제 |
| OCR           | Amazon Textract                | 영수증 이미지에서 텍스트 추출             |
| 출력 검증     | Zod 또는 JSON Schema           | LLM 출력 구조 강제                        |
| 런타임        | AWS Lambda (Node.js or Python) | 백엔드에서 호출되는 분석 함수             |
| 프롬프트 관리 | 자체 템플릿 파일               | 버전 관리 가능하도록 코드와 분리          |

---

## 개발 순서

### Phase 1 — 출력 스키마 설계 (Day 1, 최우선)

백엔드 팀이 Step Functions를 구성하려면 이 Lambda의 **입출력 구조**를 먼저 알아야 한다.
Day 1 내에 초안을 잡고 백엔드 팀과 합의한다. (구체적인 필드명과 형식은 팀 협의 후 확정)

**텍스트 파싱 Lambda 출력 구조 (초안):**

```
{
  date:        날짜 (추출 불가 시 null)
  amount:      금액 (숫자)
  category:    카테고리명 (프로젝트 설정 기준)
  description: 지출 설명
  payer:       결제자 (추출 불가 시 null)
  confidence:  신뢰도 점수 (0.0 ~ 1.0)
  needs_review: boolean
  missing_fields: 누락된 필드 목록
}
※ 실제 필드명·타입은 백엔드 팀과 협의 후 확정 (TBD)
```

**영수증 OCR Lambda 출력 구조 (초안):**

```
{
  date:         날짜
  merchant:     상호명
  items:        품목 목록 (품목명, 수량, 단가, 금액)
  total_amount: 합계
  category:     분류된 카테고리
  confidence:   신뢰도 점수
  needs_review: boolean
}
※ 실제 필드명·타입은 백엔드 팀과 협의 후 확정 (TBD)
```

### Phase 2 — 텍스트 파싱 구현 (Day 2~4, MVP 1차 핵심)

MVP 1차에서 가장 먼저 동작해야 하는 기능이다.

**프롬프트 설계 원칙:**

```
1. 출력 형식을 프롬프트 안에 명시 (JSON 구조 고정)
2. 불확실한 값은 추론하지 말고 null 반환
3. 카테고리 분류 시 프로젝트의 카테고리 목록을 컨텍스트로 주입
4. 신뢰도는 LLM이 직접 계산하지 않고, 규칙 기반으로 산정
```

**신뢰도 계산 규칙 (예시, 팀 협의 후 확정):**

```
날짜 추출 성공    → +0.3
금액 추출 성공    → +0.4
카테고리 분류 성공 → +0.2
결제자 추출 성공  → +0.1

영수증 없음(텍스트만) → needs_review = true (무조건)
신뢰도 < 0.7         → needs_review = true
예산 초과            → needs_review = true (백엔드에서 판정)
```

**테스트 케이스 작성:**

실제 입력 케이스를 최소 10개 이상 모아두고 프롬프트 수정 시마다 결과를 비교한다.

```
케이스 예시:
  "어제 행사 다과 32,000원"
  "삼겹살 158000 홍길동"
  "5/12 OO마트 영수증" (금액 없음)
  "회식비" (날짜·금액 없음)
  "2만원" (항목 없음)
```

### Phase 3 — 영수증 OCR 파이프라인 구현 (Day 5~7, MVP 2차)

```
흐름:
S3에서 이미지 읽기
  ↓
Amazon Textract 호출 (AnalyzeExpense API 권장)
  ↓
Textract 결과에서 품목·금액·날짜 추출
  ↓
Bedrock LLM으로 정제 및 카테고리 분류
  ↓
신뢰도 계산 및 결과 반환
```

**Textract AnalyzeExpense vs DetectDocumentText:**

- `DetectDocumentText`: 단순 텍스트 추출, 영수증 구조 파악 못 함
- `AnalyzeExpense`: 영수증 특화 API, 품목·금액·날짜를 구조적으로 추출 → 이걸 사용

### Phase 4 — 엑셀 양식 컬럼 매핑 (Day 8~9, MVP 2차)

관리자가 업로드한 엑셀의 컬럼 구조를 LLM이 자동으로 파악한다.

```
입력: 엑셀 파일의 헤더 행 텍스트 목록
  예: ["순번", "항목명", "수량", "단가", "금액", "비고"]

출력: 시스템 필드와 엑셀 열의 매핑 추천
  예: { item_name: "B열", quantity: "C열", ... }
  ※ 실제 필드명은 TBD

주의: 관리자가 반드시 검토 후 확정 (자동 확정 금지)
```

## 주의사항

- **Bedrock 모델 선택:** Claude Haiku는 빠르고 저렴하지만 복잡한 구조 파악이 약하다. 텍스트 파싱은 Haiku, 엑셀 컬럼 매핑은 Sonnet 사용을 권장한다.
- **프롬프트는 코드와 분리:** Lambda 코드 안에 프롬프트 문자열을 하드코딩하지 말고 별도 파일로 관리한다. 수정 시 Lambda 재배포 없이 S3나 환경변수로 교체 가능하게 한다.
- **LLM 출력은 항상 검증:** Bedrock이 반환한 JSON이 기대 스키마와 다를 수 있다. Zod나 JSON Schema로 파싱 전 검증하고, 실패 시 `needs_review: true`로 처리한다.
- **Textract 비용:** `AnalyzeExpense`는 페이지당 과금이다. MVP 테스트 시 실제 영수증 이미지를 10장 이내로 제한하며 비용을 확인한다.
- **카테고리 분류 정확도:** 카테고리 키워드 목록을 프롬프트에 명시적으로 넣어야 분류 정확도가 높아진다. 키워드 없이 LLM에게 맡기면 엉뚱한 카테고리가 나올 수 있다.
