# Team BudgetFlow 기획 총정리

## 0. 문서 개요

본 문서는 클라우드컴퓨팅 기말프로젝트 **Team BudgetFlow**의 서비스 기획 내용을 총정리한 문서이다. 프로젝트의 주제는 **멀티모달 예산 정산 자동화 메신저 봇**이며, 슬랙 메신저 환경에서 사용자가 수량, 영수증, 지출 설명 등을 전송하면 AI가 이를 분석하여 예산안 또는 지출내역서를 자동으로 작성하는 서비스를 목표로 한다.

이 문서는 다음 내용을 포함한다.

- 프로젝트 개요
- 문제 정의
- 서비스 솔루션
- 핵심 사용자 및 사용 상황
- 전체 실행 흐름
- 사용자 플로우
- 시스템 플로우
- 주요 유스케이스
- 4명 역할 분담
- AWS 기반 아키텍처
- 추천 기술 스택
- 데이터 모델 초안
- MVP 개발 범위
- 차별화 포인트
- 개발 우선순위
- 최종 서비스 방향성
- **아키텍처 설계 확정 사항** ← 고도화 검토 후 추가

## 2. 서비스 한 줄 정의

**사용자가 카카오톡 또는 슬랙에 상품 링크, 수량, 영수증 이미지, 지출 설명을 보내면 AI가 자동으로 예산 항목을 분석·분류하고, 관리자가 지정한 양식에 맞춰 예산안 또는 지출내역서 엑셀 파일을 생성하는 협업형 예산 정산 자동화 서비스**

---

## 3. 문제 정의

### 3.1 Pain Point

동아리, 학회, 학생회, 프로젝트팀, 해커톤 운영진 등 대규모 단체 행사에서는 예산안과 지출내역서를 작성하는 과정이 매우 번거롭다.

특히 다음과 같은 불편이 존재한다.

1. **예산안 작성의 번거로움**
   - 목적별 사용 가능 금액을 정해야 한다.
   - 구매할 제품의 링크, 금액, 수량을 직접 조사해야 한다.
   - 조사한 정보를 엑셀 양식에 수기로 입력해야 한다.

2. **지출내역서 작성의 번거로움**
   - 실제 구매한 품목이 기존 예산안과 일치하는지 확인해야 한다.
   - 영수증을 수집해야 한다.
   - 영수증의 물품명, 수량, 금액을 확인해야 한다.
   - 예산안에 작성된 항목과 실제 지출 항목이 맞는지 검토해야 한다.
   - 최종적으로 학교나 기관이 요구하는 양식에 맞춰 지출내역서를 작성해야 한다.

3. **팀 단위 정산에서 발생하는 문제**
   - 여러 사람이 각자 결제하고 증빙을 따로 보낸다.
   - 회계 담당자가 메신저 대화, 사진, 링크, 엑셀을 오가며 수작업으로 정리한다.
   - 영수증 누락, 금액 불일치, 항목 오분류가 자주 발생한다.
   - 마감 직전에 증빙을 다시 요청해야 하는 일이 많다.

### 3.2 문제의 본질

이 문제의 본질은 단순히 영수증을 읽는 문제가 아니라, **팀 단위 예산 집행 과정에서 발생하는 자료 수집, 항목 분류, 증빙 확인, 엑셀 양식 입력을 자동화하는 문제**이다.

따라서 본 서비스는 단순 OCR 서비스가 아니라, **메신저 기반 예산 정산 워크플로우 자동화 서비스**로 정의해야 한다.

## 4. 서비스 솔루션

### 4.1 핵심 솔루션

관리자가 예산안 양식과 목적별 사용 가능 금액을 설정하면, 사용자는 카카오톡 또는 슬랙 봇에 상품 링크, 수량, 영수증 이미지 등을 전송한다.

시스템은 이를 자동으로 분석하여 다음 작업을 수행한다.

1. 상품 링크 분석
2. 영수증 이미지 OCR 분석
3. 물품명, 금액, 수량 추출
4. 목적별 예산 항목 분류
5. 예산안 또는 지출내역서 양식에 자동 입력
6. 관리자의 검토 및 수정 지원
7. 최종 엑셀 파일 다운로드 제공

### 4.2 서비스의 핵심 가치

| 핵심 가치   | 설명                                                       |
| ----------- | ---------------------------------------------------------- |
| 입력 간소화 | 사용자는 메신저에 링크나 영수증을 보내기만 하면 된다.      |
| 자동 분석   | AI가 링크, 영수증, 텍스트에서 필요한 정보를 추출한다.      |
| 자동 분류   | 물품의 목적에 따라 예산 항목을 자동 추천한다.              |
| 양식 자동화 | 관리자가 지정한 엑셀 양식에 맞춰 자동 입력된다.            |
| 증빙 관리   | 영수증 이미지와 지출 항목을 연결하여 누락을 방지한다.      |
| 협업 최적화 | 팀원은 입력만 하고, 관리자는 검토와 승인에 집중할 수 있다. |

## 5. 주요 사용자

### 5.1 관리자

관리자는 예산안 양식과 예산 기준을 설정하고, 최종 결과를 검토하는 사용자이다.

주요 행동은 다음과 같다.

- 조직에서 사용하는 예산안 또는 지출내역서 양식 업로드
- 구분별 또는 목적별 사용 가능 금액 설정
- 자동 분석된 항목 검토
- 잘못 분류된 항목 수정
- 지출 항목 승인 또는 반려
- 최종 엑셀 파일 다운로드

### 5.2 일반 사용자

일반 사용자는 실제로 물품 링크나 영수증을 제출하는 사용자이다.

주요 행동은 다음과 같다.

- 구매 예정 물품 링크 전송
- 수량 입력
- 영수증 이미지 전송
- 지출 설명 텍스트 입력
- 봇이 보낸 확인 메시지에 응답
- 누락 증빙 요청 시 추가 제출

### 5.3 회계 담당자

회계 담당자는 관리자와 유사하지만, 특히 정산 실무를 담당한다.

주요 행동은 다음과 같다.

- 예산 항목별 지출 현황 확인
- 예산 초과 여부 확인
- 증빙 누락 항목 확인
- 지출내역서 생성
- 제출용 엑셀 파일 검토

## 6. 전체 서비스 실행 흐름

### 6.1 사용자 관점 전체 흐름

```text
관리자가 프로젝트 생성
→ 예산안/지출내역서 양식 업로드
→ 목적별 사용 가능 금액 설정
→ 카카오톡/슬랙 봇을 프로젝트 채널에 연결
→ 사용자가 링크, 수량, 영수증, 지출 설명 입력
→ AI가 입력 내용을 분석
→ 예산 항목 자동 분류
→ 관리자가 웹에서 검토 및 수정
→ 승인된 항목을 엑셀 양식에 자동 반영
→ 최종 예산안 또는 지출내역서 다운로드
```

### 6.2 시스템 관점 전체 흐름

슬랙 Webhook의 3초 응답 제한으로 인해 분석 처리는 비동기로 수행한다.

```text
[즉시 응답 단계]
슬랙 입력 수신
→ Webhook Lambda가 즉시 "접수됐습니다. 분석 중..." 응답
→ S3에 원본 데이터 저장
→ Step Functions 비동기 실행 시작

[비동기 처리 단계 - Step Functions]
→ 이미지인 경우: Textract OCR 수행
→ 텍스트인 경우: LLM 파싱
→ LLM 기반 정규화 및 카테고리 분류
→ 신뢰도 계산 및 needs_review 판정
  - 신뢰도 낮음 OR 예산 초과 OR 영수증 없음 → needs_review
  - 그 외 → auto confirmed
→ DynamoDB 저장

[완료 알림 단계]
→ 슬랙 채널에 분석 결과 메시지 전송
  - confirmed: "다음 지출로 등록됐습니다. [날짜/금액/항목]"
  - needs_review: "@관리자 검토가 필요한 항목이 있습니다. [대시보드 링크]"

[엑셀 생성 단계 - On Demand]
→ 관리자가 [정산 마감] 후 [엑셀 다운로드] 클릭
→ confirmed 항목만 엑셀 양식에 삽입 (needs_review 항목은 경고 후 제외)
→ S3에 엑셀 파일 저장
→ Presigned URL 발급 및 다운로드 링크 제공
```

## 7. 예산안 작성 Flow

### 7.1 예산안 User Flow

```text
1. 관리자가 조직에서 공통적으로 사용하는 예산안 양식을 업로드한다.
2. 관리자가 구분별 또는 목적별 사용 가능 금액을 지정한다.
3. 사용자가 구매 예정 물품의 링크와 수량을 카카오톡/슬랙 봇에 전송한다.
4. 시스템이 링크를 분석하여 상품명, 금액, 수량 등을 파악한다.
5. AI가 해당 물품을 목적에 맞게 분류한다.
6. 시스템이 관리자가 업로드한 예산안 양식에 목적, 수량, 금액, 링크 등을 입력한다.
7. 관리자 또는 사용자가 다운로드 버튼을 누르거나 봇에 다운로드를 요청한다.
8. 작성된 예산안 엑셀 파일을 다운로드한다.
```

### 7.2 예산안 Use Case

#### UC-B01. 예산안 양식 업로드

| 항목         | 내용                                                                |
| ------------ | ------------------------------------------------------------------- |
| Actor        | 관리자                                                              |
| Trigger      | 예산안 작성을 시작함                                                |
| Precondition | 관리자가 웹에 로그인되어 있음                                       |
| Main Flow    | 양식 업로드 → 시스템이 파일 저장 → 컬럼 구조 분석 → 프로젝트에 연결 |
| Result       | 해당 프로젝트의 기본 예산안 양식이 등록됨                           |

#### UC-B02. 목적별 사용 가능 금액 설정

| 항목      | 내용                                                              |
| --------- | ----------------------------------------------------------------- |
| Actor     | 관리자                                                            |
| Trigger   | 예산 범위를 설정하려 함                                           |
| Main Flow | 목적 항목 생성 → 사용 가능 금액 입력 → 키워드 또는 분류 기준 설정 |
| Result    | AI 분류 기준과 예산 한도가 설정됨                                 |

#### UC-B03. 상품 링크 기반 예산 항목 등록

| 항목             | 내용                                                                              |
| ---------------- | --------------------------------------------------------------------------------- |
| Actor            | 일반 사용자                                                                       |
| Trigger          | 구매 예정 상품 링크를 봇에 전송                                                   |
| Main Flow        | 링크 수신 → 상품 정보 추출 → 물품명/금액/수량 파악 → 목적 분류 → 예산안 항목 생성 |
| Alternative Flow | 링크 분석 실패 시 사용자가 상품명과 금액을 직접 입력                              |
| Result           | 예산안에 들어갈 구매 예정 항목이 생성됨                                           |

#### UC-B04. 예산안 엑셀 다운로드

| 항목      | 내용                                                                        |
| --------- | --------------------------------------------------------------------------- |
| Actor     | 관리자 또는 사용자                                                          |
| Trigger   | 예산안 작성 완료 후 다운로드 요청                                           |
| Main Flow | 확정된 예산 항목 조회 → 양식에 데이터 삽입 → 엑셀 생성 → 다운로드 링크 제공 |
| Result    | 제출 가능한 예산안 파일 생성                                                |

## 8. 지출내역서 작성 Flow

### 8.1 지출내역서 User Flow

```text
1. 관리자가 조직에서 사용하는 지출내역서 또는 예산안 양식을 업로드한다.
2. 관리자가 구분별 또는 목적별 사용 가능 금액을 지정한다.
3. 사용자가 영수증 등 증빙 자료를 카카오톡/슬랙 봇에 전송한다.
4. 시스템이 영수증 이미지를 OCR로 분석한다.
5. 영수증에서 물품명, 금액, 수량, 날짜, 사용처 등을 파악한다.
6. AI가 파악한 물품을 목적에 맞게 분류한다.
7. 시스템이 관리자가 업로드한 양식에 목적, 수량, 금액, 증빙 정보를 입력한다.
8. 관리자 또는 사용자가 다운로드 버튼을 누르거나 봇에 다운로드를 요청한다.
9. 작성된 지출내역서 엑셀 파일을 다운로드한다.
```

### 8.2 지출내역서 Use Case

#### UC-E01. 영수증 이미지로 지출 등록

| 항목             | 내용                                                                                         |
| ---------------- | -------------------------------------------------------------------------------------------- |
| Actor            | 일반 사용자                                                                                  |
| Trigger          | 카카오톡/슬랙에 영수증 이미지 업로드                                                         |
| Precondition     | 프로젝트와 예산 항목이 생성되어 있음                                                         |
| Main Flow        | 이미지 업로드 → OCR 분석 → 물품명/금액/수량 추출 → LLM 정리 → 항목 추천 → 사용자 확인 → 저장 |
| Alternative Flow | OCR 신뢰도 낮음 → 검토 필요 상태로 저장                                                      |
| Result           | 지출 내역과 영수증 이미지가 연결됨                                                           |

#### UC-E02. 텍스트로 지출 등록

| 항목             | 내용                                                                  |
| ---------------- | --------------------------------------------------------------------- |
| Actor            | 일반 사용자                                                           |
| Trigger          | 사용자가 “어제 행사 다과 32,000원”과 같은 문장을 입력                 |
| Main Flow        | 텍스트 수신 → 날짜/금액/항목 추출 → 예산 항목 추천 → 확인 메시지 전송 |
| Alternative Flow | 날짜나 사용처가 없으면 오늘 날짜로 추정하거나 사용자에게 확인 요청    |
| Result           | 영수증이 없어도 임시 지출 항목으로 등록됨                             |

#### UC-E03. 예산안과 지출내역 비교

| 항목             | 내용                                                                     |
| ---------------- | ------------------------------------------------------------------------ |
| Actor            | 시스템, 관리자                                                           |
| Trigger          | 실제 지출 항목 등록                                                      |
| Main Flow        | 지출 항목 조회 → 기존 예산안 항목과 비교 → 품목/수량/금액 일치 여부 확인 |
| Alternative Flow | 금액 초과 또는 품목 불일치 시 검토 필요 표시                             |
| Result           | 예산안 대비 실제 지출의 일치 여부 확인                                   |

#### UC-E04. 증빙 누락 알림

| 항목      | 내용                                                             |
| --------- | ---------------------------------------------------------------- |
| Actor     | 시스템                                                           |
| Trigger   | 지출 내역에는 금액이 있으나 영수증이 없음                        |
| Main Flow | 증빙 누락 탐지 → 해당 사용자에게 봇 알림 → 영수증 추가 제출 요청 |
| Result    | 정산 마감 전 증빙 누락 방지                                      |

#### UC-E05. 지출내역서 엑셀 다운로드

| 항목      | 내용                                                                                     |
| --------- | ---------------------------------------------------------------------------------------- |
| Actor     | 관리자 또는 회계 담당자                                                                  |
| Trigger   | 최종 지출내역서 생성 요청                                                                |
| Main Flow | 승인된 지출 항목 조회 → 증빙 연결 상태 확인 → 엑셀 양식에 자동 입력 → 다운로드 링크 제공 |
| Result    | 제출 가능한 지출내역서 파일 생성                                                         |

- 텍스트 지출 입력 LLM 파싱
- 카테고리 자동 분류 및 신뢰도 계산

- 로그인 (Cognito)

- 슬랙 채널 연결 설정
- 지출 목록 화면 (5초 폴링, needs_review 뱃지)
- needs_review 항목 검토·수정 화면
- 정산 마감 버튼 + 엑셀 다운로드 (경고 포함)

## 18. 기존 서비스와 차별화

### 18.1 일반 영수증 OCR 서비스와의 차이

| 구분      | 일반 OCR 서비스    | BudgetFlow                            |
| --------- | ------------------ | ------------------------------------- |
| 목적      | 영수증 텍스트 추출 | 예산안/지출내역서 자동 작성           |
| 입력      | 주로 영수증 이미지 | 링크, 수량, 영수증, 텍스트, 엑셀 양식 |
| 사용자    | 개인 사용자 중심   | 팀, 동아리, 학회, 행사 운영진 중심    |
| 결과      | 텍스트 데이터      | 제출 가능한 엑셀 파일                 |
| 핵심 기능 | 문자 인식          | 분류, 검토, 증빙 연결, 양식 반영      |
| 협업 기능 | 약함               | 팀 단위 정산 흐름 지원                |

### 18.2 삼쩜삼/네이버 클로바 OCR과의 차별점

BudgetFlow는 OCR 자체가 핵심이 아니라 OCR 이후의 정산 업무를 자동화한다.

차별점은 다음과 같다.

1. **메신저 기반 입력**
   - 별도 앱을 켜지 않고 카카오톡/슬랙에서 바로 입력한다.

2. **예산안과 지출내역서 연결**
   - 구매 예정 링크와 실제 영수증을 연결하여 예산안과 지출내역을 비교할 수 있다.

3. **조직별 양식 반영**
   - 학교, 학과, 기관마다 다른 엑셀 양식을 업로드하고 해당 양식에 맞춰 결과를 생성한다.

4. **목적별 예산 분류**
   - 단순 금액 추출이 아니라 식비, 다과비, 홍보비, 물품비 등 목적별로 자동 분류한다.

5. **증빙 누락 관리**
   - 영수증이 없는 지출을 자동으로 탐지하고 제출을 요청한다.

6. **회계 담당자 검토 흐름**
   - AI 결과를 바로 확정하지 않고 관리자가 검토하고 승인할 수 있다.

## 21. 오류 및 예외 처리

### 21.1 OCR 실패

상황:

- 영수증 이미지가 흐림
- 글자가 잘림
- 금액 인식이 부정확함

처리 방식:

```text
OCR 실패 또는 신뢰도 낮음
→ needs_review 상태로 저장
→ 사용자에게 재업로드 요청
→ 관리자가 수동 입력 가능
```

### 21.2 링크 분석 실패

상황:

- 쇼핑몰 페이지 접근 불가
- 로그인 필요
- 상품 가격이 동적으로 로딩됨

처리 방식:

```text
링크 분석 실패
→ 상품명/금액/수량 직접 입력 요청
→ 링크는 참고 자료로 저장
```

### 21.3 예산 초과

상황:

- 특정 목적 항목의 사용 가능 금액을 초과함

처리 방식:

```text
예산 초과 감지
→ 관리자 검토 필요 표시
→ 봇 알림 전송
→ 승인 전까지 엑셀 반영 제한
```

### 21.4 증빙 누락

상황:

- 텍스트로 지출 등록했으나 영수증이 없음

처리 방식:

```text
증빙 없음 상태 저장
→ 마감 전 알림 발송
→ 영수증 추가 제출 요청
```

## 22. 보안 및 권한 관리

### 22.1 권한 유형

| 권한        | 설명                                                      |
| ----------- | --------------------------------------------------------- |
| 관리자      | 프로젝트 생성, 양식 업로드, 항목 승인, 엑셀 다운로드 가능 |
| 회계 담당자 | 지출 검토, 수정, 다운로드 가능                            |

### 22.2 보안 고려사항

- 영수증 이미지와 엑셀 파일은 S3에 저장한다.
- 다운로드 링크는 Presigned URL로 발급하고 만료 시간을 둔다.
- 프로젝트별 접근 권한을 분리한다.
- 개인정보 또는 결제 정보가 포함될 수 있으므로 외부 공개를 제한한다.
- 봇에서 받은 파일은 프로젝트와 사용자 정보에 연결하여 저장한다.

## 23. 발표 시 강조할 포인트

발표에서는 다음 흐름을 중심으로 설명하면 좋다.

```text
문제: 예산안과 지출내역서 작성이 번거롭다.
해결: 메신저에 링크와 영수증만 보내면 AI가 자동 정리한다.
핵심: 단순 OCR이 아니라 조직별 양식에 맞춘 정산 자동화이다.
구현: Slack/Kakao Bot + AWS 서버리스 + Textract + Bedrock + Web Dashboard.
결과: 관리자는 검토만 하고, 최종 엑셀 파일을 다운로드한다.
```

### 23.1 시연 시나리오 예시

```text
1. 관리자가 웹에서 프로젝트를 생성한다.
2. 예산안 양식을 업로드한다.
3. 식비 300,000원, 다과비 100,000원 등 예산 항목을 설정한다.
4. 사용자가 슬랙에 “어제 행사 다과 32,000원”이라고 입력한다.
5. 봇이 “다과비 32,000원으로 등록할까요?”라고 확인한다.
6. 사용자가 등록 버튼을 누른다.
7. 관리자가 웹에서 해당 항목을 확인하고 승인한다.
8. 엑셀 다운로드 버튼을 누른다.
9. 완성된 지출내역서가 다운로드된다.
```

## 24. 최종 서비스 방향성

BudgetFlow는 단순히 영수증을 읽는 OCR 서비스가 아니다. 이 서비스의 핵심은 **예산안 작성, 실제 지출 등록, 증빙 수집, 항목 분류, 양식 입력, 최종 엑셀 다운로드까지 이어지는 정산 업무 전체 흐름을 자동화하는 것**이다.

따라서 개발 과정에서는 다음 기준을 우선해야 한다.

1. 사용자가 메신저에서 쉽게 입력할 수 있어야 한다.
2. AI가 예산 항목을 정확하게 추천해야 한다.
3. 관리자가 웹에서 쉽게 검토하고 수정할 수 있어야 한다.
4. 최종 결과물이 실제 제출 가능한 엑셀 파일이어야 한다.
5. 자동화 결과를 그대로 확정하지 않고, 검토와 승인 과정을 포함해야 한다.

최종적으로 본 프로젝트는 다음과 같은 구조로 완성된다.

```text
프론트엔드:
Next.js 기반 관리자 대시보드

백엔드:
AWS Lambda + API Gateway + Step Functions 기반 서버리스 API

LLM/OCR:
Amazon Textract + Amazon Bedrock 기반 입력 분석 및 분류 엔진

봇:
Slack Bolt 우선 구현, 이후 Kakao i Open Builder 또는 카카오톡/카카오워크 확장
```

## 25. 최종 요약

Team BudgetFlow의 서비스는 카카오톡/슬랙 봇을 통해 사용자가 링크, 수량, 영수증 등을 전송하면 AI가 이를 분석하여 예산안 또는 지출내역서를 자동 작성하는 서비스이다.

이를 위해 다음과 같이 역할을 나눈다.

- **봇 담당자**는 사용자 입력과 알림 흐름을 구현한다.
- **백엔드 담당자**는 API, DB, 파일 저장, 엑셀 생성 등 시스템 중심 구조를 구현한다.
- **LLM/OCR 담당자**는 영수증·링크·텍스트 분석과 예산 항목 분류를 구현한다.
- **프론트엔드 담당자**는 관리자 검토 화면과 엑셀 다운로드 기능을 구현한다.

네 역할은 독립적으로 개발할 수 있지만, 최종적으로는 다음 하나의 흐름으로 연결된다.

```text
메신저 입력
→ AI 분석
→ 예산 항목 분류
→ 관리자 검토
→ 엑셀 자동 생성
→ 제출용 파일 다운로드
```

따라서 이 프로젝트의 핵심 경쟁력은 **AI 인식률 자체가 아니라, 예산 정산 업무를 팀 단위 협업 흐름에 맞게 자동화한다는 점**에 있다.

## 26. 아키텍처 설계 확정 사항

기획 고도화 검토를 통해 확정된 설계 결정 사항을 기록한다.

### 26.1 구조 및 권한

| 항목          | 결정                                        | 이유                                                                |
| ------------- | ------------------------------------------- | ------------------------------------------------------------------- |
| 조직 구조     | Organization → Project 계층 (멀티테넌시)    | 동아리는 매 학기 행사를 반복하므로 조직 단위로 양식·카테고리 재사용 |
| 메신저 채널   | 슬랙 채널 1:1 프로젝트 매핑                 | 채널이 곧 행사 공간, 팀원 공유 흐름 유지                            |
| 카카오톡      | MVP 제외, 슬랙만 구현                       | Kakao i 오픈빌더 구조상 그룹 공유 흐름 구현 어려움                  |
| 사용자 온보딩 | Slack Workspace 연동, 팀원 별도 가입 불필요 | 동아리원 전원 가입 요구 시 이탈률 급증                              |
| 관리자 인증   | Amazon Cognito                              | 웹 대시보드 접근 권한 관리                                          |

### 26.2 분석 및 처리

| 항목           | 결정                                         | 이유                                                            |
| -------------- | -------------------------------------------- | --------------------------------------------------------------- |
| 엑셀 양식 파싱 | LLM 자동 컬럼 매핑 추천 (X-B)                | 관리자가 직접 드래그앤드롭하는 UX보다 자동 추천이 매끄러움      |
| 매핑 확정 방식 | 관리자 검토 필수 (R-1)                       | 금액 열 오매핑 시 엑셀에 잘못된 데이터 삽입, 조용한 오류 방지   |
| 슬랙 응답 방식 | 즉시 접수 확인 + 비동기 분석 결과 전송 (T-1) | 슬랙 Webhook 3초 제한, OCR+LLM 파이프라인은 10~30초 소요        |
| 예산 매핑      | 카테고리 단위 소진 추적 (M-1)                | 품목 1:1 매핑은 LLM 오매핑 시 데이터 오염, 실제 운영과도 불일치 |
| 상품 링크 분석 | MVP 제외 (V-1)                               | 쇼핑몰 동적 렌더링·로그인 장벽으로 실패 케이스 多, 발표 리스크  |

### 26.3 승인 및 상태 관리

| 항목                   | 결정                                                              | 이유                                                           |
| ---------------------- | ----------------------------------------------------------------- | -------------------------------------------------------------- |
| needs_review 조건      | AI 신뢰도 낮음 OR 예산 초과 OR 영수증 없음(무조건)                | 증빙 없는 항목은 지출증빙 목적상 반드시 관리자 확인 필요       |
| 자동 승인              | 위 3가지 조건 미해당 시 auto confirmed                            | 관리자가 정상 케이스를 건별로 승인하면 운영 부담 과다          |
| 예산안·지출내역서 관계 | 완전 독립 생성 가능 (D-2)                                         | 동아리는 예산안 없이 먼저 집행하고 나중에 정산하는 경우가 다수 |
| 엑셀 생성 정책         | needs_review 있어도 생성 가능, confirmed 항목만 포함 + 경고 (E-2) | 마감 당일 늦은 영수증 때문에 전체 제출이 막히는 상황 방지      |
| 프로젝트 마감          | 관리자 수동 [정산 마감] 클릭 (L-2)                                | 동아리 행사 특성상 늦은 제출이 많아 자동 마감 시 운영 혼란     |

### 26.4 운영 및 UX

| 항목          | 결정                                            | 이유                                                                        |
| ------------- | ----------------------------------------------- | --------------------------------------------------------------------------- |
| 대시보드 갱신 | 5초 폴링 (U-1)                                  | TanStack Query refetchInterval으로 구현 단순, 관리자 대시보드는 채팅이 아님 |
| 관리자 알림   | 슬랙 채널 멘션 (N-1) + 대시보드 뱃지 (N-2) 병행 | 별도 알림 채널은 온보딩 장벽, MVP에서는 채널 멘션으로 충분                  |
